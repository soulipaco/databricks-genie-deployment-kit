# Skill: pipeline_update_playbook

## Trigger
- "update the playbook"
- "update playbook content"
- "refresh the playbook"
- "update scenarios for X"
- "add an action card"
- "update action plans playbook"
- "regenerate the playbook PDF"

## Purpose

Update the content of an existing domain's playbook (scenarios, action cards, example plans,
KPI descriptions, interpretation rules) and propagate the change through the full chain:
edit blueprint → regenerate PDF → upload → re-sync VS index.

The most common mistake is editing `playbook_blueprint.yml` without re-generating the PDF,
or re-generating without re-syncing the Vector Search index — this skill prevents both.

Use `pipeline_add_domain` instead when adding a brand-new domain that does not yet exist.

## Inputs

| Input | Source | Required | Notes |
|---|---|---|---|
| `domain_key` | User | Yes | The key under `domains:` in `playbook_blueprint.yml` to update |
| `what_to_update` | User | Yes | One or more of: `scenarios`, `action_cards`, `example_plans`, `kpis_covered`, `interpretation_rules`, `intro_text` |
| `new_content` | User | Yes | The actual content changes — scenarios to add/replace, new action cards, etc. |

## Steps

### Step 1 — Identify the domain

Read `pipeline_playbook_generator/config/playbook_blueprint.yml`.

Find the domain block matching `domain_key` under `domains:`.

If the key is not found:
- List all existing domain keys to the user
- Ask whether they meant a different key, or if this is a new domain (→ redirect to `pipeline_add_domain`)

Read `pipeline/pipeline_config.yml` and confirm that `domain_key`'s `question_category`
appears in all three sections: `playbooks.sources`, `questions.seeds`, `system_prompts`.
If any are missing, warn the user before proceeding.

### Step 2 — Edit `playbook_blueprint.yml`

Apply the content changes to the identified domain block.

**Scenarios:** Replace the `scenarios:` list or append new scenario entries.
Each scenario needs: `letter`, `title`, `symptoms`, `likely_causes`,
`action_plan_recommendations` (with `heading` + `actions`), `measurement_targets`.

**Action cards:** Replace the `action_cards:` list or append new entries.
Each card needs: `title`, `trigger`, `actions`, `expected_impact`.

**Example plans:** Replace or append under `example_plans:` in the domain block.
Each plan needs: `number`, `title`, `observed_trend`, `plan`, `success_metrics`.

**KPI descriptions / interpretation rules / intro_text:** Edit the relevant fields directly.

Do not change `display_name`, `question_category`, or `playbook_title` unless explicitly asked —
changing `question_category` would break the cross-notebook contract and requires `pipeline_add_domain` logic.

### Step 3 — Regenerate PDFs (local machine)

```bash
python pipeline_playbook_generator/generate_playbook.py --kit-format genie_kit
```

Confirm the updated PDF appears in `pipeline_playbook_generator/generated/pdfs/`
with a newer modification time than before.

If the generator fails:
- Check that the domain key still partially matches a `metadata/columns/*.yml` filename
- Check for YAML syntax errors in the edited blueprint

### Step 4 — Upload updated PDF (if source_type: volume)

Read `pipeline/pipeline_config.yml → playbooks.source_type`.

If `source_type == "volume"`:
```bash
python setup/upload_pdfs.py
```
This will overwrite the existing PDF in the UC Volume with the updated version.

If `source_type == "local"`, skip — the notebook reads from the repo path directly.

### Step 5 — Re-sync Vector Search index (Databricks — instruct user)

The existing chunks for this domain will be updated via MERGE (same `chunk_id` = overwrite).

Instruct the user:
> In Databricks, open `setup/vector_search.py` and run it. It will re-chunk the updated
> PDF, MERGE the updated rows into `{prefix}_playbook_chunks`, and sync the index.

After running, validate that the index sync completes:
```sql
-- Verify chunks were updated (check for recent modification timestamps)
SELECT question_category, COUNT(*) AS chunk_count
FROM {catalog}.{schema}.{table_prefix}_playbook_chunks
WHERE question_category = '<question_category>'
GROUP BY question_category;
```

> **Important:** Until the VS index finishes syncing, action plans generated by
> `action_plans.py` will still retrieve the old playbook content.
> Wait for the index status to return to ONLINE before running the pipeline.

### Step 6 — Optionally update the LLM system prompt

If the content change warrants a revised expert prompt (e.g. new KPIs were added,
or the response structure should change), update `pipeline_config.yml → system_prompts`
for the affected `question_category`.

This takes effect on the next run of `pipeline/action_plans.py` — no Databricks
setup step is needed.

### Step 7 — Report to user

Summarize:
- Which domain was updated
- Which sections changed
- Whether PDF was regenerated and uploaded
- Reminder to run `setup/vector_search.py` if not yet done
- Reminder to check VS index is ONLINE before relying on new retrieval results

## Outputs

- Updated `pipeline_playbook_generator/config/playbook_blueprint.yml`
- Updated `pipeline_playbook_generator/generated/pdfs/<pdf_filename>`
- UC Volume (if `source_type: volume`) — Updated PDF uploaded
- `{prefix}_playbook_chunks` — Updated after running `setup/vector_search.py`

## Validation

- [ ] `playbook_blueprint.yml` domain block reflects all intended changes
- [ ] Updated PDF exists in `generated/pdfs/` with a newer modification time
- [ ] If `source_type: volume`: upload completed without error
- [ ] `setup/vector_search.py` has been run in Databricks after the PDF update
- [ ] VS index sync completed and status is ONLINE
- [ ] `question_category` value was NOT changed (if it was, run `pipeline_add_domain` logic)
